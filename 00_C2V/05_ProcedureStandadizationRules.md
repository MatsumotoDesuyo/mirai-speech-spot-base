# C2V Engine 手順の標準化ルール(Procedure Standardization Rules)

## 1. 目的

標準手順（プロシージャ）を統一された形式で記述することにより、人間とAIの双方にとっての可読性、実行可能性、およびメンテナンス性を最大化する。

## 2. ルールの適用範囲

`Ignition_Sequence.md`や`Define_Role.md`など、AIによるガイドが想定される全ての標準手順書に適用する。

## 3. 必須記載項目

全ての手順書は、以下の項目を必ず含んでいなければならない。

### 3.1. ヘッダー情報
手順書の冒頭には、以下の情報を記載する。
-   **Procedure ID**: 手順を一意に識別するID（例: P001）
-   **Version**: バージョン番号（例: 1.0）
-   **Last Updated**: 最終更新日

### 3.2. 手順の目的 (Purpose)
この手順が「なぜ存在するのか」「何を達成するためのものか」を1〜2文で明確に記述する。

### 3.3. 手順のトリガー (Trigger)
この手順が「いつ」「どのような状況で」開始されるべきかを記述する。
-   *例：「新しいロールを定義する必要性が生じたとき」*

### 3.4. ステップ (Steps)
実行すべき具体的な手順を、番号付きリストで時系列に記述する。各ステップは、単一のアクションになるように簡潔に記述する。

### 3.5. 成果物 (Artifacts)
この手順を完了した際に、作成または更新されるべき最終的な成果物を定義する。

## 4. ファイル命名規則

手順書のファイル名は、`[ProcedureName]_Procedure.md`の形式で統一する。
-   *例：`Define_Role_Procedure.md`, `Onboarding_Procedure.md`*

## 5. バージョン管理

手順書の変更は、必ずGitのコミットログに変更内容の要約を記録する。軽微な修正はパッチバージョン（例: 1.0 -> 1.1）、大きな変更はマイナーバージョン（例: 1.1 -> 2.0）を更新する。
