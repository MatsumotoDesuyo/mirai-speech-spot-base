# ツール開発ポリシー (Tool Development Policy)

## 1. このドキュメントの目的
このドキュメントは、C2Vオーケストレーションプロジェクトにおける、内部用ツールを開発する際のプロセス、ルール、品質基準を定義します。

本ポリシーの目的は、C2Vの基本思想を尊重しつつ、**本流のプロダクト開発とは異なるツール開発の文脈に最適化された、一貫性のある開発プロセスを確立すること**です。

## 2. 「ツール」の定義
本ポリシーにおける「ツール」とは、以下の通り定義します。

> C2Vの価値創造・価値届出において、**本質的思考（ビジョン策定や意思決定）以外の、定型的・反復的な作業を自動化または支援する**ためのソフトウェア。

## 3. 開発プロセス
ツールの開発は、本流のプロダクト開発プロセスを以下の通り**簡略化**して適用します。

| プロセス階層 | ツール開発での適用方針 | 詳細 |
| :--- | :--- | :--- |
| **価値戦略層** (`01_value_strategy`) | 🔻 **省略** | ツールの目的は本流プロダクトの開発支援にあり、独立したビジョンは不要なため。 |
| **アーキテクチャ・ルール層** | 🟡 **簡略化** | ツールに特化したアーキテクチャの指定がなければ、**デフォルトのアーキテクチャ** (`tools/docs/default_tool_architecture.md`) を採用する。`README.md`の作成は必須。重要な技術選定を行う場合はADRを作成する。 |
| **仕様策定** | ✅ **維持** | ツールが「何をすべきか」を定義する壁打ちはこの段階で行う。仕様書を `tools/[ツール名]/docs/specification.md` として保存する。 |
| **タスク層** | ✅ **維持** | 仕様書を、下流のAIが実装できるレベルの具体的な指示に落とし込む。 |

## 4. ドキュメント管理
全てのツール関連ドキュメントは、以下の場所に配置します。

- **ツール全般**: `tools/docs/`
- **ツール固有**: `tools/[ツール名]/docs/`

各ツールのディレクトリには、最低限以下のドキュメントを含める必要があります。
- `README.md`: ツールの目的、使用方法、技術スタックを記述。
- `docs/specification.md`: 仕様策定で作成された、ツールの仕様書。

## 5. 品質基準
ツールの品質基準は、その重要度と複雑性に応じて2つのレベルに分類します。

### レベル1：個人用スクリプト
- **定義**: 開発者個人が使用する、単機能でシンプルなツール。
- **品質基準**:
  - **テストコード**: 推奨だが、**必須ではない**。
  - **ドキュメント**: `README.md`が整備されていることは**必須**。

### レベル2：共有ワークフローツール
- **定義**: CI/CDパイプラインへの組み込みや、他のAI・開発者からの利用が想定される、より重要で複雑なツール。
- **品質基準**:
  - このレベルのツールの必要性が生じた**その時点**で、より厳格な品質基準（テストコードの必須化など）を**改めて定義する**。

## 6. ポリシーの改訂
このポリシーは、プロジェクトの進捗や新たな知見に応じて改訂される、生きたドキュメントです。
