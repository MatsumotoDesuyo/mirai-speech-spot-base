# 開発ワークフローとルール (Development Workflow)

本プロジェクトでは、複数の開発者が効率的かつコンフリクトなく開発を進められるよう、以下の開発ワークフローと規約を採用しています。

## 1. ブランチ戦略: GitHub Flow

本プロジェクトは、シンプルでアジャイルな開発に適した **GitHub Flow** を採用しています。複雑な管理（Git Flow等）は避け、常に本番環境へデプロイ可能な状態を維持します。

1. **`main` ブランチは常にデプロイ可能**: `main` ブランチは常に安定しており、いつでも本番環境にデプロイできる状態を保ちます。
2. **作業用ブランチを作成する**: 新機能の追加やバグ修正を行う際は、必ず `main` から新しい機能ブランチを作成します。
   * **ブランチ名の例**:
     * `feature/add-map-pin` (新機能)
     * `bugfix/fix-login-error` (バグ修正)
     * `docs/update-readme` (ドキュメント更新)
3. **ローカルでコミットする**: 変更を加え、後述の「Conventional Commits」に従ってコミットメッセージを記述します。
4. **Pull Request (PR) を作成する**: 作業が完了したら、`main` ブランチに向けてPRを作成します。PR上でコードレビューや議論を行います。
5. **マージする**: レビューが承認され、CIのテストをパスしたら、`main` にマージされます。

## 2. コミットメッセージ規約: Conventional Commits

コミットの意図を明確にし、他開発者や将来の自分にとって履歴を追いやすくするため、**[Conventional Commits](https://www.conventionalcommits.org/ja/v1.0.0/)** を採用しています。

### 基本フォーマット
```text
<type>[optional scope]: <description>
```

### Type（種類）
コミットの性質に応じて、以下のプレフィックスを使用してください。

* **`feat`**: 新機能の追加 (Feature)
* **`fix`**: バグの修正 (Bug fix)
* **`docs`**: ドキュメントの変更のみ (Documentation)
* **`style`**: コードの動作に影響しないフォーマットの変更 (スペース、セミコロン欠落など)
* **`refactor`**: バグ修正や機能追加を含まないコードの書き換え (Refactoring)
* **`perf`**: パフォーマンスを向上させるコード変更 (Performance)
* **`test`**: テストの追加や修正 (Tests)
* **`chore`**: ビルドプロセスやツール、ライブラリの変更 (Chores)

### 記入例
* `feat(map): 演説スポットのピンの色をスコアに応じて変更する機能を追加`
* `fix(auth): パスコード入力エラー時にアプリがクラッシュする問題を修正`
* `docs: READMEにデザイナー向けのポータルリンクを追加`

## 3. Pull Request のルール

* PRを作成する際は、必ず **PRテンプレート (`PULL_REQUEST_TEMPLATE.md`)** の内容に従って記述してください。（※テンプレート準備中）
* その変更が「どのIssueを解決するものか」「ミッションにどう貢献するか」を必ず明記してください。ミッションから逸脱したPRはマージされません。
